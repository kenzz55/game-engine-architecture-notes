## 1. 도구가 중요한 이유

게임 엔진 개발은 단순히 C++ 코드를 작성하는 작업만으로 끝나지 않는다.  
대규모 게임 프로젝트에서는 소스 코드, 애셋, 설정 파일, 빌드 결과물, 디버깅 정보, 성능 측정 데이터까지 수많은 산출물이 계속 변경된다.

따라서 게임 개발에서 도구는 생산성을 높이는 보조 수단이 아니라, 프로젝트의 규모가 커질수록 개발 자체를 가능하게 해주는 기반 시스템에 가깝다.

2장에서 다루는 주요 도구는 다음과 같다.

- 버전 컨트롤 시스템
- 컴파일러
- 링커
- IDE
- 디버거
- 프로파일러
- 메모리 검사 도구
- 비교/병합 도구
- 헥스 에디터

게임 엔진 프로그래머는 코드를 작성하는 사람인 동시에, 빌드·디버깅·성능 측정·협업 환경을 이해하는 개발자여야 한다.

## 2. 버전 컨트롤

버전 컨트롤 시스템은 파일의 변경 이력을 관리하는 도구다.  
게임 프로젝트에서는 여러 프로그래머, 아티스트, 기획자가 동시에 작업하기 때문에 누가 어떤 파일을 언제, 왜 수정했는지 추적할 수 있어야 한다.

버전 컨트롤을 쓰는 이유는 다음과 같다.

- 중심 저장소 또는 원격 저장소를 통해 프로젝트 파일을 공유한다.
- 파일의 변경 이력을 기록한다.
- 문제가 생겼을 때 이전 버전으로 되돌릴 수 있다.
- 여러 개발자가 동시에 작업할 수 있다.
- 브랜치를 통해 기능 개발, 버그 수정, 출시 버전을 분리할 수 있다.
- 코드 리뷰와 변경 추적이 가능하다.

게임 개발에서는 코드뿐 아니라 텍스처, 모델, 사운드, 애니메이션, 맵 데이터 같은 대형 바이너리 파일도 함께 관리해야 한다. 그래서 일반적인 소프트웨어 프로젝트보다 버전 컨트롤의 중요성이 더 크다.

### 널리 쓰이는 버전 컨트롤 시스템

역사적으로 사용된 대표적인 버전 컨트롤 시스템은 다음과 같다.

- SCCS
- RCS
- CVS
- Subversion
- Git
- Perforce
- NxN Alienbrain
- ClearCase

현재는 Git이 가장 널리 쓰이지만, 대형 게임 개발에서는 Perforce 계열 도구도 여전히 많이 사용된다.

### Git

Git은 분산 버전 컨트롤 시스템이다. 각 개발자가 전체 저장소 이력을 로컬에 가지고 있기 때문에 브랜치 생성, 커밋, diff 확인 같은 작업이 빠르다.

Git은 개인 프로젝트, 오픈소스, 코드 중심 프로젝트에 매우 적합하다.  
다만 대형 바이너리 파일이 많아지는 게임 프로젝트에서는 Git LFS 같은 추가 도구나 별도의 애셋 관리 전략이 필요할 수 있다.

### Subversion과 TortoiseSVN

Subversion은 중앙 집중식 버전 컨트롤 시스템이다. 모든 변경 이력이 중앙 저장소에 있고, 사용자는 필요한 파일을 체크아웃해서 작업한다.

TortoiseSVN은 Subversion을 윈도우 탐색기와 통합해 사용할 수 있게 해주는 대표적인 GUI 클라이언트다.

### Perforce

Perforce는 대규모 프로젝트와 대형 바이너리 파일 관리에 강점이 있다.  
게임 프로젝트는 코드보다 애셋 용량이 훨씬 커질 수 있고, 바이너리 파일은 텍스트처럼 쉽게 병합하기 어렵다.

그래서 AAA 게임 스튜디오에서는 Perforce 같은 중앙 집중식 시스템을 많이 사용한다.  
특히 누가 어떤 파일을 체크아웃했는지 명확히 파악하고, 병합하기 어려운 파일을 잠그는 방식으로 충돌을 예방할 수 있다는 점이 중요하다.

## 3. 파일 버전, 업데이트, 커밋

버전 컨트롤에서 기본이 되는 작업은 업데이트, 수정, 커밋이다.

### 업데이트

업데이트는 저장소의 최신 변경 사항을 내 작업 공간으로 가져오는 작업이다.

```bash
git pull
```

### 커밋

커밋은 내가 수정한 내용을 저장소에 기록하는 작업이다.

```bash
git add .
git commit -m "docs: add chapter 2 tools notes"
```

### 체크아웃

체크아웃은 저장소의 특정 파일, 특정 브랜치, 특정 버전을 로컬 작업 공간으로 가져오는 작업이다.

```bash
git checkout main
```

### 체크인

체크인은 중앙 집중식 시스템에서 내가 수정한 파일을 저장소에 반영하는 작업을 의미한다.  
Git에서는 보통 `commit`과 `push`를 분리해서 생각한다.

### diff

diff는 두 버전 사이에 어떤 내용이 달라졌는지 비교하는 작업이다.

```bash
git diff
```

실무에서는 커밋 단위를 작게 유지하는 것이 중요하다.  
예를 들어 `이동 시스템 수정`, `UI 수정`, `프로파일링 로그 추가`를 한 커밋에 모두 넣으면 나중에 변경 이유를 추적하기 어렵다. 반대로 기능 단위로 커밋하면 문제가 생겼을 때 어떤 변경이 원인인지 찾기 쉽다.

## 4. 다중 체크아웃, 브랜치, 병합

여러 사람이 같은 파일을 동시에 수정할 수 있는 구조를 다중 체크아웃이라고 볼 수 있다.

텍스트 파일은 두 사람이 서로 다른 위치를 수정했다면 자동으로 병합될 수 있다.  
하지만 같은 줄이나 같은 영역을 수정하면 충돌이 발생한다.

### 브랜치

브랜치는 독립적인 작업 흐름을 만들기 위한 기능이다.

예를 들어 다음과 같이 분리할 수 있다.

- `main`: 안정 버전
- `develop`: 개발 통합 브랜치
- `feature/chat-system`: 채팅 시스템 작업
- `fix/movement-jitter`: 이동 보간 버그 수정

브랜치를 쓰면 새로운 기능을 실험하면서도 안정적인 메인 브랜치를 보호할 수 있다.

### merge

merge는 서로 다른 브랜치의 변경 사항을 합치는 작업이다.

```bash
git merge feature/chat-system
```

### three-way merge

3자 병합은 다음 세 가지를 비교한다.

1. 공통 조상 버전
2. 내 로컬 수정 버전
3. 상대방 또는 서버의 수정 버전

단순히 두 파일만 비교하는 것이 아니라, 두 변경이 어디에서 갈라졌는지를 기준으로 병합하기 때문에 자동 병합 가능성이 높아진다.

예를 들어 `Foo.cpp version 4`에서 두 사람이 동시에 작업했다고 하자.

```text
        Foo.cpp (joe_b)
       /
Foo.cpp (version 4)
       \
        Foo.cpp (suzie_q)
```

만약 `suzie_q`가 먼저 체크인해서 `version 5`가 되었고, `joe_b`가 나중에 체크인하려면 다음 비교가 필요하다.

```text
1. 공통 조상: Foo.cpp version 4
2. 서버 최신: Foo.cpp version 5
3. 내 로컬: Foo.cpp joe_b
```

결국 병합 도구는 `version 4`를 기준으로 `version 5`의 변경과 `joe_b`의 변경을 비교해서 최종 결과를 만든다.

### 게임 개발에서 병합이 어려운 이유

게임 개발에서는 텍스트 파일뿐 아니라 바이너리 애셋도 많다.

예를 들어 다음 파일은 사람이 의미 있게 병합하기 어렵다.

- `.uasset`
- `.umap`
- 이미지 파일
- 사운드 파일
- 바이너리 애니메이션 데이터
- DCC 툴 원본 파일

이런 파일은 자동 병합보다 **파일 잠금**, **작업 구역 분리**, **명확한 소유권 규칙**이 중요하다.

## 5. 컴파일러, 링커, IDE

C++ 게임 엔진 개발에서 코드는 바로 실행 파일이 되지 않는다.  
소스 코드는 전처리, 컴파일, 링크 단계를 거쳐 실행 파일 또는 라이브러리가 된다.

### 컴파일러

컴파일러는 C++ 소스 코드를 기계어에 가까운 오브젝트 파일로 변환한다.

### 링커

링커는 여러 오브젝트 파일과 라이브러리를 묶어 최종 실행 파일 또는 DLL을 만든다.

### IDE

IDE는 편집, 빌드, 디버깅, 프로젝트 관리 기능을 제공하는 통합 개발 환경이다.

Visual Studio 같은 IDE는 단순한 코드 편집기를 넘어, 솔루션/프로젝트 관리, 빌드 설정, 디버거, 프로파일러와 연결된다.

게임 엔진 개발에서는 여러 프로젝트가 서로 의존하는 경우가 많기 때문에 IDE의 프로젝트 구조를 이해하는 것이 중요하다.

## 6. 소스 파일, 헤더 파일, 번역 단위

C++에서 `.cpp`, `.cc`, `.cxx` 같은 파일은 보통 소스 파일이라고 부른다.

헤더 파일은 타입 선언, 함수 선언, 클래스 선언, 상수, 템플릿 등을 여러 소스 파일에서 공유하기 위해 사용한다.

중요한 점은 컴파일러가 헤더 파일을 독립적인 컴파일 단위로 직접 컴파일하는 것이 아니라는 점이다.  
전처리 단계에서 `#include` 구문은 해당 헤더 파일의 내용으로 치환된다. 그 결과 하나의 소스 파일과 그 안에 포함된 헤더 내용이 합쳐져 하나의 번역 단위가 된다.

```cpp
// Player.cpp
#include "Player.h"
#include "Inventory.h"
```

위 예시에서 컴파일러는 단순히 `Player.cpp`만 보는 것이 아니라, 전처리 결과로 `Player.h`, `Inventory.h`의 내용이 포함된 하나의 큰 파일을 컴파일한다. 이것이 번역 단위다.

### 게임 엔진에서 중요한 이유

게임 엔진처럼 코드베이스가 큰 프로젝트에서는 헤더 하나를 수정했을 때 수백 개의 `.cpp`가 다시 컴파일될 수 있다.

따라서 다음을 신경 써야 한다.

- 불필요한 `#include` 줄이기
- 전방 선언 사용
- 헤더 의존성 최소화
- 공통 헤더 남용 방지
- PCH 사용 시 장단점 이해

## 7. 라이브러리, 실행 파일, DLL

각 번역 단위가 컴파일되면 오브젝트 파일이 만들어진다.  
하지만 오브젝트 파일만으로는 프로그램을 실행할 수 없다.

여러 오브젝트 파일을 서로 연결하고, 외부 심볼을 해결해야 한다. 이 작업을 링킹이라고 한다.

링커가 하는 일은 다음과 같다.

- 여러 오브젝트 파일을 하나로 묶는다.
- 함수 호출과 전역 변수 참조를 실제 주소에 연결한다.
- 정적 라이브러리 또는 동적 라이브러리를 연결한다.
- 최종 실행 파일 또는 DLL을 만든다.

### 정적 라이브러리

정적 라이브러리는 링크 시점에 실행 파일 안으로 포함된다.

장점:

- 배포가 단순하다.
- 실행 파일 하나만으로 동작하기 쉽다.
- 런타임 DLL 누락 문제가 적다.

단점:

- 실행 파일 크기가 커질 수 있다.
- 라이브러리 수정 시 실행 파일을 다시 링크해야 한다.

### DLL

DLL은 Dynamic Link Library의 약자다.  
실행 파일과 분리되어 존재하다가 실행 시점에 운영체제에 의해 로드된다.

DLL의 장점은 다음과 같다.

- 여러 실행 파일이 같은 DLL을 공유할 수 있다.
- 특정 기능을 모듈 단위로 교체할 수 있다.
- 플러그인 구조를 만들 수 있다.
- 실행 파일 크기를 줄일 수 있다.
- 엔진과 에디터, 툴 사이에서 공통 코드를 공유할 수 있다.

다만 DLL은 배포와 버전 관리가 중요하다.  
실행 파일이 기대하는 DLL 버전과 실제 배포된 DLL 버전이 다르면 런타임 오류가 발생할 수 있다.

## 8. 프로젝트와 솔루션

Visual Studio에서 솔루션은 여러 프로젝트를 담는 상위 단위다.  
프로젝트는 하나의 실행 파일, 라이브러리, DLL, 테스트 모듈 등을 만들기 위한 빌드 단위다.

게임 프로젝트에서는 보통 다음과 같은 프로젝트 구성이 가능하다.

```text
GameSolution
├─ GameClient
├─ GameServer
├─ EngineCore
├─ Renderer
├─ Physics
├─ EditorTool
├─ AssetPipeline
└─ UnitTests
```

각 프로젝트는 서로 의존할 수 있다.

예를 들어:

- `GameClient`는 `EngineCore`와 `Renderer`에 의존할 수 있다.
- `GameServer`는 `EngineCore`와 `Networking`에 의존할 수 있다.
- `EditorTool`은 `EngineCore`와 `AssetPipeline`에 의존할 수 있다.

이런 의존 관계를 잘못 설계하면 빌드 시간이 길어지고 순환 참조 문제가 생길 수 있다.

## 9. 빌드 설정

빌드 설정은 같은 코드라도 어떤 목적으로 빌드할지 결정하는 옵션 묶음이다.

대표적으로 Debug와 Release가 있다.

### Debug 빌드

Debug 빌드는 개발 중 디버깅을 쉽게 하기 위한 설정이다.

특징:

- 디버그 정보 포함
- 최적화 꺼짐 또는 최소화
- assertion 활성화
- 검증 코드 활성화
- 실행 속도는 느림
- 버그를 찾기 쉬움

### Release 빌드

Release 빌드는 최종 배포에 가까운 설정이다.

특징:

- 최적화 활성화
- 디버그 정보 축소 또는 제외
- 불필요한 검증 코드 제거
- 실행 속도 빠름
- 디버깅 어려움

### 게임 개발에서의 추가 빌드 구성

게임 개발에서는 Debug와 Release만으로 부족한 경우가 많다.

그래서 다음과 같은 빌드 구성을 따로 두기도 한다.

| 빌드 구성 | 목적 |
|---|---|
| Debug | 디버깅용. 느리지만 검증이 많음 |
| Development | 개발 중 실행 확인용. 일부 최적화와 로그 사용 |
| Test/Profile | 실제 성능 측정용 |
| Shipping/Release | 최종 배포용. 디버그 기능 최소화 |

게임 개발에서는 “실행 가능한 성능”과 “디버깅 가능성” 사이의 균형을 맞추는 것이 중요하다.

## 10. 전처리기 설정

전처리기는 컴파일 전에 실행되는 단계다.

대표적인 전처리 작업은 다음과 같다.

- `#include` 파일 포함
- `#define` 매크로 치환
- 조건부 컴파일
- 빌드별 심볼 처리

예를 들어 `_DEBUG` 같은 심볼은 디버그 빌드에서만 특정 코드를 활성화하는 데 사용될 수 있다.

```cpp
#ifdef _DEBUG
    CheckMemoryLeak();
#endif
```

이런 방식은 개발 중에는 유용하지만, 전처리기 조건이 많아지면 코드 흐름을 이해하기 어려워질 수 있다.  
따라서 빌드 설정별 분기를 남발하지 않는 것이 좋다.

---

## 11. 컴파일러 옵션과 최적화

컴파일러 옵션은 코드 생성 방식에 직접 영향을 준다.

예를 들어 다음을 결정한다.

- 디버그 정보를 포함할지
- 최적화를 켤지
- 경고 수준을 어떻게 할지
- 런타임 라이브러리를 어떤 방식으로 연결할지
- 인라인 확장을 어느 정도 허용할지
- SIMD 명령어를 사용할지
- 예외 처리와 RTTI를 사용할지

### 로컬 최적화

로컬 최적화는 제한된 범위 안에서 수행되는 최적화다.

예시:

- 상수 폴딩
- 상수 전파
- 죽은 코드 제거
- 간단한 연산 치환
- 루프 펼치기
- 코드 인라인화
- 명령어 재배열

### 글로벌 최적화

글로벌 최적화는 함수나 번역 단위의 경계를 넘어 더 넓은 범위에서 수행되는 최적화다.

특히 링커 타임 최적화는 여러 오브젝트 파일을 링크하는 시점에 전체 프로그램 정보를 활용해 최적화를 수행한다.

### 인라인 함수

인라인 함수는 함수 호출 비용을 줄이기 위해 함수 본문을 호출 지점에 펼치는 최적화다.

하지만 `inline` 키워드를 붙였다고 반드시 인라인되는 것은 아니다. 최종 결정은 컴파일러가 한다. 함수가 너무 크거나, 주소가 필요하거나, 최적화 조건에 맞지 않으면 인라인되지 않을 수 있다.

## 12. LTO와 PGO

### LTO

LTO는 Link-Time Optimization의 약자다.  
링크 단계에서 더 넓은 범위의 최적화를 수행하는 방식이다.

일반 컴파일은 번역 단위 단위로 이루어지기 때문에 다른 `.cpp` 파일 내부의 정보를 충분히 알기 어렵다.  
LTO는 링크 시점에 전체 프로그램 정보를 더 많이 활용할 수 있게 한다.

장점:

- 함수 경계를 넘는 최적화 가능
- 불필요한 코드 제거 가능
- 더 적극적인 인라인화 가능

단점:

- 링크 시간이 길어진다.
- 빌드 파이프라인이 복잡해질 수 있다.
- 디버깅이 어려워질 수 있다.

### PGO

PGO는 Profile-Guided Optimization의 약자다.  
실제 실행 데이터를 바탕으로 최적화하는 방식이다.

PGO의 일반적인 흐름은 다음과 같다.

```text
1. 계측 빌드 생성
2. 실제 사용 시나리오 실행
3. 프로파일 데이터 수집
4. 수집 데이터 기반 최적화 빌드 생성
```

게임에서는 PGO가 특히 의미 있을 수 있다.  
예를 들어 로그인 화면에서만 쓰는 코드보다 매 프레임 실행되는 이동, 애니메이션, 렌더링, 물리 코드가 더 중요하다. PGO는 이런 실제 실행 패턴을 반영해 자주 쓰는 경로를 더 잘 최적화할 수 있다.

다만 LTO와 PGO는 빌드 시간을 증가시킬 수 있다.  
따라서 개발 중 매번 사용하기보다는 성능 측정용 빌드나 출시 후보 빌드에서 사용하는 것이 현실적이다.

## 13. 코드 디버깅

디버깅은 프로그램의 실행 흐름과 상태를 추적해 버그 원인을 찾는 과정이다.

IDE 디버거는 중단점, 한 줄씩 실행, 콜스택 확인, 변수 조사, 메모리 확인 같은 기능을 제공한다.

대표적인 디버깅 기능은 다음과 같다.

### 중단점

특정 코드 위치에서 실행을 멈춘다.

### Step Over

현재 줄을 실행하되 함수 내부로 들어가지 않는다.

### Step Into

함수 내부로 들어간다.

### Step Out

현재 함수 실행을 끝내고 호출한 곳으로 돌아간다.

### 콜스택

현재 함수가 어떤 호출 경로로 실행되었는지 보여준다.

### 조사식 창

변수나 표현식 값을 확인한다.

### 데이터 중단점

특정 메모리 주소의 값이 바뀔 때 실행을 멈춘다.  
메모리 오염 문제를 찾을 때 특히 유용하다.

### 게임 개발에서 디버깅이 어려운 이유

게임 개발에서는 디버거만으로 모든 문제를 찾기 어렵다.

대표적인 이유:

- 프레임 단위로 계속 실행된다.
- 멀티스레딩으로 실행 순서가 매번 달라질 수 있다.
- 네트워크 지연과 패킷 순서 문제가 있다.
- 물리와 애니메이션은 시간 흐름에 민감하다.
- 렌더링 버그는 시각화 도구가 함께 필요하다.
- 디버거로 멈추는 순간 재현 조건이 깨질 수 있다.

따라서 로그, 리플레이, 프로파일러, 시각화 도구를 함께 사용해야 한다.

---

## 14. 최적화된 빌드 디버깅

릴리스 빌드에서만 발생하는 버그는 게임 개발에서 자주 만나는 문제다.

디버그 빌드에서는 문제가 없는데 릴리스 빌드에서만 크래시가 나거나 동작이 달라질 수 있다.

대표적인 원인은 다음과 같다.

- 초기화하지 않은 변수
- 메모리 오염
- 정의되지 않은 동작
- 구조체 패딩 차이
- 컴파일러 최적화로 인한 코드 재배치
- 인라인 함수로 인한 콜스택 변화
- 타이밍 차이로 드러나는 멀티스레딩 버그

최적화된 빌드는 변수 값이 레지스터에 있거나, 코드가 재배열되거나, 함수가 인라인되어 소스 코드와 실행 흐름이 1:1로 대응하지 않을 수 있다.

그래서 다음 능력이 중요하다.

- 디스어셈블리 읽기
- 레지스터 값 해석
- 주소만 보고 객체나 변수 추적
- 메모리 창에서 구조체 내용 확인
- 정적 변수와 전역 변수 활용
- 크래시 덤프 분석

게임 엔진 프로그래머는 단순히 C++ 코드만 읽는 것이 아니라, 최적화된 결과물이 실제로 어떻게 실행되는지 이해해야 한다.

---

## 15. 프로파일링 툴

프로파일링은 프로그램의 성능 병목을 찾는 과정이다.

게임에서는 평균 FPS만 보는 것으로는 부족하다.  
어느 함수가 느린지, 어느 프레임에서 튀는지, CPU와 GPU 중 어디가 병목인지, 메모리 할당이 언제 발생하는지 확인해야 한다.

### 파레토 법칙

파레토 법칙은 전체 결과의 대부분이 소수의 원인에서 나온다는 경험적 법칙이다.  
성능 최적화에서도 비슷한 상황이 자주 발생한다.

전체 프레임 시간의 대부분을 몇몇 함수나 시스템이 차지하는 경우가 많다.

최적화는 다음 순서로 접근하는 것이 좋다.

```text
1. 먼저 측정한다.
2. 가장 큰 병목을 찾는다.
3. 수정한다.
4. 다시 측정한다.
5. 성능이 실제로 좋아졌는지 확인한다.
```

### 함수가 시간을 많이 잡아먹는 두 가지 이유

함수가 시간을 많이 잡아먹는 이유는 크게 두 가지다.

1. 함수 자체가 한 번 실행될 때 오래 걸린다.
2. 함수 자체는 짧지만 수천, 수만 번 호출된다.

게임 루프에서는 두 번째 경우가 매우 중요하다.  
예를 들어 1회 실행 시간이 짧은 함수라도 매 프레임 수천 개의 객체에 대해 호출된다면 전체 성능 병목이 될 수 있다.

### 통계 방식 프로파일러

통계 방식 프로파일러는 일정 시간마다 프로그램의 실행 위치를 샘플링한다.

장점:

- 프로그램 실행에 주는 영향이 비교적 작다.
- 전체적인 병목을 찾기 좋다.
- 실제 실행 환경에 가까운 데이터를 얻을 수 있다.

단점:

- 짧게 실행되는 함수는 놓칠 수 있다.
- 정확한 호출 횟수를 알기 어렵다.

### 인스트루먼트 방식 프로파일러

인스트루먼트 방식 프로파일러는 함수 진입과 종료 지점에 계측 코드를 삽입해 정확한 호출 횟수와 실행 시간을 측정한다.

장점:

- 호출 횟수와 실행 시간을 자세히 알 수 있다.
- 함수별 비용을 정확히 분석하기 좋다.

단점:

- 프로그램 실행 속도가 크게 느려질 수 있다.
- 계측 자체가 성능 특성을 바꿀 수 있다.

프로파일링에서 중요한 태도는 추측하지 않는 것이다.  
“아마 이 함수가 느릴 것이다”라고 생각하고 최적화하는 것이 아니라, 실제 데이터를 보고 가장 큰 병목부터 해결해야 한다.

## 16. 메모리 누수와 메모리 오염 감지

### 메모리 누수

메모리 누수는 할당한 메모리를 더 이상 사용하지 않는데 해제하지 못하는 문제다.

오랫동안 실행되는 게임 서버나 게임 클라이언트에서는 작은 누수도 시간이 지나면 큰 문제가 된다.

### 메모리 오염

메모리 오염은 할당 범위를 벗어난 쓰기, 해제된 메모리 접근, 이중 해제, 잘못된 포인터 사용 등으로 인해 메모리 내용이 손상되는 문제다.

메모리 오염은 발생 지점과 크래시 지점이 다를 수 있어 찾기 어렵다.

대표적인 메모리 문제는 다음과 같다.

- 메모리 누수
- use-after-free
- double free
- buffer overflow
- uninitialized memory read
- invalid pointer access

### 대표적인 메모리 검사 도구

- Rational Purify
- Insure++
- Valgrind
- AddressSanitizer
- Visual Studio Diagnostic Tools
- CRT Debug Heap

Valgrind의 Memcheck 같은 도구는 C/C++ 프로그램의 잘못된 메모리 접근과 누수를 찾는 데 사용된다.

게임 개발에서는 직접 만든 메모리 할당기, 오브젝트 풀, 참조 카운팅, 스마트 포인터, 핸들 시스템이 자주 등장한다.  
이 구조들은 성능상 유리하지만 버그가 생기면 찾기 어려우므로 메모리 검증 도구와 로깅 체계를 함께 갖춰야 한다.

## 17. 기타 툴

대규모 개발에서는 IDE와 디버거 외에도 다양한 보조 도구가 필요하다.

### 비교 툴

비교 툴은 두 파일의 차이를 시각적으로 보여준다.

예시:

- WinMerge
- Beyond Compare
- Meld
- Visual Studio diff
- Git diff tool

### 3자 병합 툴

3자 병합 툴은 다음 세 버전을 함께 보여주며 충돌을 해결하게 해준다.

- 공통 조상
- 내 수정

게임 개발에서는 다음 상황에서 유용하다.

- 세이브 파일 구조 확인
- 리소스 파일 헤더 확인
- 엔디안 문제 확인
- 손상된 파일 분석

### 게임 개발에서 추가로 중요한 도구

게임 엔진 개발에서는 다음 도구들도 중요하다.

- 로그 뷰어
- 크래시 덤프 분석 도구
- 패킷 캡처 도구
- 빌드 자동화 도구
- 정적 분석 도구
- 포맷터와 린터
- 에셋 검증 도구
- 리플레이/녹화 도구

특히 게임 서버 개발에서는 네트워크 패킷 분석, 로그 추적, 부하 테스트, 프로파일링 도구가 중요하다.

- 재현 가능한가?
- 어떤 입력에서 문제가 생기는가?
- 문제가 발생하기 직전 상태를 기록할 수 있는가?

## 18. 게임 엔진 개발 관점 핵심 정리

2장의 핵심은 도구를 단순한 부가 기능으로 보지 말라는 것이다.

게임 개발은 규모가 커질수록 사람이 직접 기억하고 관리할 수 없는 복잡도를 갖게 된다.  
도구는 이 복잡도를 관리하기 위한 시스템이다.

게임 엔진 프로그래머에게 중요한 도구 이해는 다음과 같다.

- 버전 컨트롤을 통해 변경 이력과 협업을 관리한다.
- 컴파일러와 링커 구조를 이해해 빌드 오류와 링크 오류를 해결한다.
- 빌드 설정을 이해해 Debug, Development, Release의 차이를 구분한다.
- 디버거를 통해 실행 흐름과 상태를 분석한다.
- 프로파일러를 통해 추측이 아니라 데이터 기반으로 최적화한다.
- 메모리 검사 도구를 통해 누수와 오염을 조기에 잡는다.
- 비교/병합/헥스 에디터 같은 보조 도구로 파일 단위 문제를 해결한다.
- 게임 서버나 엔진 포트폴리오에서는 “동작한다”보다 “측정하고 분석했다”가 더 중요하다.

즉, 게임 엔진 개발자는 코드를 작성하는 사람인 동시에, 빌드·디버깅·성능 측정·협업 환경을 이해하는 개발자여야 한다.
